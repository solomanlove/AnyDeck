# Skill: doc-git-commit (Git 规范化提交)

## 概述
本技能用于规范化项目的 Git 提交流程。当需要提交代码时，AI 助手必须调取本技能文档，确保提交信息与历史提交风格完全一致，并清晰总结实际解决的问题。

## Git 提交规范流程

### 0. 暂存区边界确认
提交代码前必须先执行 `git status --short` 和 `git diff --cached --stat`，区分“本次要提交的文件”和“用户或其他流程已经暂存的文件”。

- 如果用户明确说“提交代码”但没有要求整理暂存区，只提交当前已经待提交且属于本次任务的文件。
- 其他已暂存但不属于本次任务的文件不要擅自 `git restore --staged`、不要改动、不要纳入本次 commit；必须保留原有暂存状态。
- 如果暂存区里混有本次任务文件和无关文件，优先使用 `git commit <file...> -m "..."` 或在提交前向用户确认边界，避免移动用户已经暂存的内容。
- 只有用户明确要求“取消暂存”“整理暂存区”“只保留这些文件暂存”时，才允许对其他暂存文件执行 `git restore --staged`。

### 1. 风格比对与分析
在生成 commit message 之前，必须执行以下命令读取最近的提交历史，分析风格：
```bash
git log -n 10 --oneline
```
- **前缀分类**：严格使用 `feat:`（新功能）、`fix:`（问题修复）、`refactor:`（代码重构）、`docs:`（文档更新）、`build:`（编译/构建配置调整）等。
- **语言**：解释部分使用中文。
- **条理性**：如果单次提交包含多项修改，必须使用 `1、... 2、...` 进行分项总结，或者在 body 中分行说明。

### 2. 总结实际问题与修改
严禁写无意义的、过于简短的提交（如 `fix` 或 `update`），或者只罗列修改了哪些文件。必须说明：
- **实际遇到的问题**是什么（例如：在子 Isolate 投屏窗口中，由于心跳停用导致 `deviceOnlineProvider` 始终为 false，导致尺寸轮询定时器被立即销毁，导致触控坐标以物理分辨率换算而被 Android 端忽略）。
- **具体解决方案**是什么（例如：子窗口下 `deviceOnlineProvider` 改用投屏 Session 状态判断；并在 notifier 中添加 scrcpy 进程 exitCode 监听以自动执行清理）。

### 3. 生成与提交命令
确保 commit 消息精炼但信息完整。示例命令：
```bash
git commit -m "fix: 1、修复多窗口 Isolate 下 deviceOnlineProvider 始终为 false 导致投屏点击失效的问题；2、优化 scrcpy 服务进程退出时的生命周期自动清理"
```
